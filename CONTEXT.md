# Dossier

A daily, mostly-automated pipeline that discovers, vets, and prepares personalised connection requests targeting tech recruiters on LinkedIn. The human performs the final send.

## Language

**Pipeline**:
The end-to-end workflow from search result to a human-dispatched connection request.
_Avoid_: Workflow, process, job

**Profile**:
A LinkedIn member whose details have been captured and are progressing through the Pipeline.
_Avoid_: Contact, lead, person, candidate

**Member URN**:
LinkedIn's stable internal identifier for a Profile (e.g. `urn:li:fsd_profile:ACoAA…`). Does not change when a vanity URL changes. The canonical dedup key.
_Avoid_: Profile ID, LinkedIn ID, vanity URL (when used as an identifier)

**Snippet metadata**:
The limited Profile data visible in a Sales Navigator search result without visiting the Profile page: name, title, headline, company.
_Avoid_: Search result, preview, card data

**View budget**:
The daily ceiling on Profile visits imposed by Sales Navigator. The most constrained resource in the Pipeline — each visit is both a detection signal and a quota unit.
_Avoid_: Rate limit, quota, API limit

**Profile visit**:
A Playwright-driven load of a Profile's LinkedIn page. Consumes one unit of the View budget.
_Avoid_: Scrape, fetch, crawl

**Three-axis classification**:
The LLM-driven scoring model applied to each Profile. The three axes are: role relevance (is this person a recruiter/HR/headhunter?), activity signal (are they currently active and hiring?), and tech alignment (is their company's stack close to mine?).
_Avoid_: Score, rating, ranking

**Hard gate**:
A binary filter that permanently drops a Profile from the current Pipeline run. Two exist: the Pre-filter and the Classifier gate. A Profile dropped by either never proceeds to the next stage.
_Avoid_: Filter, threshold check, disqualifier

**Pre-filter**:
Hard gate 1. A regex/keyword match on Snippet metadata title only. Runs before any Profile visit to protect the View budget.
_Avoid_: Title filter, keyword filter

**Classifier gate**:
Hard gate 2. A threshold check on Three-axis classification scores. Runs after LLM analysis.
_Avoid_: Score threshold, LLM filter

**Draft message**:
The personalised LinkedIn connection note generated by the LLM during classification. Editable by the human before dispatch. 300 characters max (LinkedIn limit).
_Avoid_: Message, note, outreach, template

**Review queue**:
The set of Profiles that have passed both Hard gates and are awaiting human review on the dashboard.
_Avoid_: Inbox, queue, pending list

**Outcome**:
The recorded result of a dispatched connection request: `accepted`, `replied`, `meeting`, or `ghosted`.
_Avoid_: Result, response, status

**Mode file**:
A markdown file containing the system prompt and context for one Pipeline stage. Loaded by Node services at startup. One file per stage, one shared context file.
_Avoid_: Prompt file, system prompt, instruction file

**Liveness check**:
A recurring job that detects Profiles that have become stale: changed roles, gone private, or already connected.
_Avoid_: Staleness check, profile refresh

**Search executor**:
The CLI command that runs saved Sales Navigator searches, paginates results, and stores URLs with Snippet metadata. No Profile visits.
_Avoid_: Scraper, crawler, discovery job

**Profile fetcher**:
The CLI command that visits Profile URLs that passed the Pre-filter, downloads full page HTML, and stores it for parsing. Consumes View budget.
_Avoid_: Scraper, crawler, profile scraper

## Relationships

- A **Profile** is identified throughout the system by its **Member URN**, not its URL
- A **Profile** moves through the **Pipeline** in order: Search executor → Pre-filter → Profile fetcher → Worker → Classifier gate → Review queue
- **Snippet metadata** is the only Profile data available before a **Profile visit** occurs
- The **Pre-filter** acts on **Snippet metadata** to protect the **View budget**
- A **Profile visit** consumes one unit of the **View budget**
- The Worker produces a **Three-axis classification** and a **Draft message** for every Profile that passes the **Pre-filter**
- The **Classifier gate** drops Profiles below threshold before they reach the **Review queue**
- An **Outcome** is recorded for each Profile after a human dispatches its **Draft message**
- A **Liveness check** can re-flag a Profile in the **Review queue** as stale before dispatch

## Example dialogue

> **Dev:** "Should we re-fetch the Profile page before the human sends the connection request, in case things changed?"
> **Domain expert:** "No — that wastes View budget and risks detection. The Liveness check handles staleness asynchronously. The Profile visit already happened; we only re-visit if the Liveness check flags a role change."

> **Dev:** "We found the same person twice in search results under different URLs. Do we dedup on URL?"
> **Domain expert:** "No — always on Member URN. The Pre-filter will see the same URN twice and the dedup table will drop the second one before any Profile visit occurs."
